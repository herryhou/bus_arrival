## 1. 核心架構總覽（含圖）

### 1.1 二層架構：控制層 vs 估計層

```
┌───────────────────────────────────────────────────────────────┐
│                         CONTROL LAYER                         │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐          │
│  │ SystemMode  │   │   tick()    │   │  Recovery   │          │
│  │ (Normal/    │◄─ │ orchestrator│──►│   Module    │          │
│  │  OffRoute/  │   │             │   │  (pure fn)  │          │
│  │  Recovering)│   └──────┬──────┘   └─────────────┘          │
│  └─────────────┘          │                                   │
│         ▲                 │ EstimationOutput                  │
│         │                 │ (z_gps_cm, s_cm, divergence_d2)   │
│         │                 ▼                                   │
└─────────┼─────────────────────────────────────────────────────┘
          │
          │ EstimationInput (GPS, route_data)
          │
┌─────────┴─────────────────────────────────────────────────────┐
│                       ESTIMATION LAYER                        │
│  ┌─────────────────────────────────────────────────────────┐  │
│  │  GPS → Map Match → Project → Kalman → DR → Output       │  │
│  │         (純函數無控制層狀態)                               │  │
│  └─────────────────────────────────────────────────────────┘  │
│  Internal State: KalmanState, DrState (opaque to control)     │
└───────────────────────────────────────────────────────────────┘
```

**Why 分離？**  
- 估計層可獨立測試（相同 GPS → 相同輸出）。  
- 控制層統一使用估計層信號觸發模式轉換，避免競爭條件。  
- 恢復邏輯集中於純函數，易於驗證。

### 1.2 全系統資料流（離線 + 線上）

```
┌─────────────────────────────────────────────────────────────────┐
│                    OFFLINE (PC / Preprocessor)                  │
│  Raw Polyline + Stops                                           │
│      ↓                                                          │
│  ┌─────────────────────────────────────────────┐                │
│  │ ① Polyline Simplification (Douglas-Peucker)│                 │
│  │ ② Route Linearization (s in cm)            │                 │
│  │ ③ Spatial Grid Index (100m cells)          │                 │
│  │ ④ Stop Corridor & DP Projection            │                 │
│  └─────────────────────────────────────────────┘                │
│                         ↓                                       │
│                  route_data.bin (~12KB)                         │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ Include in Flash (XIP)
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    RUNTIME (Pico 2, 1Hz)                        │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ GPS NMEA → Heading‑Constrained Map Match → Project →      │  │
│  │ Speed/Monotonic Filter → Kalman → Dead Reckoning          │  │
│  └───────────────────────────────────────────────────────────┘  │
│                              ↓                                  │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ Stop Corridor Filter → Probability Model → State Machine  │  │
│  │                → Arrival / Announce Events                │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 問題與挑戰

| 問題 | 描述 | 系統解決方案 |
|------|------|-------------|
| GPS 漂移 (±30m) | 單純距離閾值誤報 | 廊道過濾 + 概率模型 (F1..F4) |
| GPS 跳點 (>100m) | 錯誤跳到下一站 | 速度/單調性過濾 + 脫離路線檢測 |
| GPS 斷訊 (隧道) | 位置中斷 | 航位推算 (DR) 持續 10 秒 |
| 近距離站點 (<120m) | 第二站 pre‑corridor 被壓縮 | 自適應廊道分配 + 移除 dwell 權重 |
| 脫離路線（繞路） | 產生大量錯誤事件 | 5 秒確認 → 位置凍結 → 重入時 Snap |

## 3. 功能需求

### 3.1 離線預處理

| ID | 功能 | Why |
|----|------|-----|
| P1 | 路線線性化為一維 `s` (cm) | 將問題降維，後續全為純量運算 |
| P2 | 空間格網索引 (100m cells) | 地圖匹配從 O(N) 降為 O(k)，保 1Hz 即時 |
| P3 | 預算路段係數 (dx, dy, seg_len_mm, heading) | Runtime 零幾何重算，無 sqrt/sin/cos |
| P4 | 站點廊道預計算 (pre 80m / post 40m) + 重疊截斷 | 提供 10‑15 秒語音預告，減少相鄰站干擾 |
| P5 | 近距站點 (<120m) 動態廊道分配 (55% pre / 10% gap / 35% post) | 確保第二站有足夠 early detection 空間 |

### 3.2 二層控制/估計分離

| ID | 功能 | Why |
|----|------|-----|
| C1 | 估計層純函數：`estimate(GPS, route_data, state) -> EstimationOutput` | 可獨立測試，保證確定性 |
| C2 | 控制層狀態機：Normal / OffRoute / Recovering | 統一管理模式轉換，單一職責 |
| C3 | 統一觸發：僅使用 `divergence_d2` 與 `z_gps_cm` 轉換模式 | 避免控制層依賴內部估計狀態 |
| C4 | 每個 tick 最多一次模式轉換 | 防止競爭條件，狀態確定 |

### 3.3 脫離路線（偏移道路）檢測與恢復

| ID | 功能 | Why |
|----|------|-----|
| D1 | 連續 5 秒 divergence > 50m → OffRoute | 5 秒遲滯避免瞬時雜訊誤觸 |
| D2 | OffRoute 期間凍結 `s_cm` 與 `v̂` | 防止錯誤位置累積，抑制虛假到站事件 |
| D3 | 重回路線且位移 > 50m → Recovering，執行 GPS snap | 直接跳到投影位置，繞過未經站點，避免漸進追趕誤報 |
| D4 | 重回路線且位移 ≤ 50m → Normal | 小幅漂移不需完整恢復 |
| D5 | Recovering 逾時 30 秒 → 幾何最近站點回退 | 確保系統不永久卡死 |

**脫離路線狀態機圖**:

```
                    divergence > 50m
                    for 5 ticks
    ┌──────────┐     ┌──────────────┐     divergence ≤ 50m    ┌───────────┐
    │  Normal  │────►│   OffRoute   │     for 2 ticks         │ Recovering│
    │(tracking)│     │  (frozen)    │────────────────────────►│ (search)  │
    └──────────┘     └──────┬───────┘                         └─────┬─────┘
           ▲                │                                       │
           │                │ displacement > 50m                    │ success
           │                │ (re-entry)                            │ or timeout
           │                ▼                                       │
           │          ┌─────────────┐                               │
           └──────────│  Snap to    │◄──────────────────────────────┘
                      │  z_reentry  │
                      └─────────────┘
```

### 3.4 到站判定與語音播報

| ID | 功能 | Why |
|----|------|-----|
| S1 | 廊道過濾：`s_cm` 進入 [corridor_start, corridor_end] 才啟動偵測 | 排除遠方漂移，自然提供提前量 |
| S2 | 四特徵概率融合：F1(GPS距離) + F2(低速) + F3(Kalman距離) + F4(dwell) | 單一指標不可靠，融合後達 97% 準確率 |
| S3 | 近距自適應權重：下一站 <120m 時移除 F4 | 短站距公車可能快過，dwell 無參考價值 |
| S4 | 語音播報：`s_cm >= corridor_start_cm` 且未播報該站 → 立即觸發 | 80m 前置自然滿足 10‑15 秒預告 |
| S5 | 單向狀態機 + 單次到站規則 (`announced` 永不重置) | 徹底防止 GPS 倒退或迴圈造成的重複報站 |

## 4. 非功能需求

| ID | 需求 | 目標 | Why |
|----|------|------|-----|
| N1 | CPU 使用率 (1 Hz) | < 8% (≤ 1.5 ms) | 保留給其他任務 (4G/CAN) |
| N2 | SRAM (runtime) | < 1 KB | 極低佔用，利於 RTOS 整合 |
| N3 | Flash (路線資料) | ≤ 12 KB | 可儲存數十條路線，無需外部儲存 |
| N4 | 準確率 (到站) | ≥ 97% | 核心 KPI |
| N5 | 斷訊容忍 | 10 秒 DR，誤差 < 150 m | 隧道場景可用 |
| N6 | 脫離路線恢復 | 重回路線後 2 秒內正常偵測 | 繞路不影響後續站點 |
| N7 | 啟動暖機 | 3 秒內抑制偵測，Kalman 收斂 | 避免冷啟動誤報 |

## 5. 成功標準

1. **功能驗證**  
   - `ty225_normal` → 所有站點正確觸發到站/離站。  
   - `ty225_shortcut` → 跳過站點不觸發任何事件。  
   - `ty225_short_detour` → 脫離路線 5 秒後位置凍結，重入時直接 snap 至前方站點，中間站點全數跳過。

2. **性能驗證**  
   - 單次 GPS 更新完整 pipeline < 1.5 ms (RP2350)。  
   - `route_data.bin` 尺寸 ≤ 12 KB。

3. **可靠度驗證**  
   - 模擬斷訊 10 秒 → 恢復後 2 秒內同步正確站點。  
   - 模擬 GPS 倒退 (>50m) → 無重複到站事件。  
   - 長時間運行 (8 小時) → 無記憶體洩漏或狀態偏移。
